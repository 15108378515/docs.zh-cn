## <a name="increasing-middle-tier-client-performance"></a><span data-ttu-id="c1601-104">提高中间层客户端应用程序的性能</span><span class="sxs-lookup"><span data-stu-id="c1601-104">Increasing Middle-Tier Client Performance</span></span>  
 <span data-ttu-id="c1601-105">与以前的通信技术，例如使用的 Web 服务相比[!INCLUDE[vstecasp](../../../../includes/vstecasp-md.md)]，创建 WCF 客户端实例会更复杂由于 WCF 丰富功能集。</span><span class="sxs-lookup"><span data-stu-id="c1601-105">Compared to previous communications technologies, such as Web services using [!INCLUDE[vstecasp](../../../../includes/vstecasp-md.md)], the creation of a WCF client instance can be more complex due to the rich feature set of WCF.</span></span> <span data-ttu-id="c1601-106">例如，打开 <xref:System.ServiceModel.ChannelFactory%601> 对象时，该对象会与服务建立一个安全会话，该过程将增加客户端实例的启动时间。</span><span class="sxs-lookup"><span data-stu-id="c1601-106">For example, when a <xref:System.ServiceModel.ChannelFactory%601> object is opened it can establish a secure session with the service, a procedure that increases the startup time for the client instance.</span></span> <span data-ttu-id="c1601-107">通常情况下，这些附加功能不会影响客户端应用程序极大地由于 WCF 客户端执行几次调用后将关闭。</span><span class="sxs-lookup"><span data-stu-id="c1601-107">Typically, these additional feature capabilities do not affect client applications greatly since the WCF client makes several calls, and then closes.</span></span>  
  
 <span data-ttu-id="c1601-108">中间层客户端应用程序，但是，可以快速创建多个 WCF 客户端对象并，因此，实现提高的初始化要求。</span><span class="sxs-lookup"><span data-stu-id="c1601-108">Middle-tier client applications, however, can create many WCF client objects quickly and, as a result, experience increased initialization requirements.</span></span> <span data-ttu-id="c1601-109">调用服务时，有两种主要的方法可以提高中间层应用程序的性能：</span><span class="sxs-lookup"><span data-stu-id="c1601-109">There are two main approaches to increasing the performance of middle-tier applications when calling services:</span></span>  
  
-   <span data-ttu-id="c1601-110">缓存的 WCF 客户端对象并将其重复用于后续调用在可能的情况。</span><span class="sxs-lookup"><span data-stu-id="c1601-110">Cache the WCF client object and reuse it for subsequent calls where possible.</span></span>  
  
-   <span data-ttu-id="c1601-111">创建<xref:System.ServiceModel.ChannelFactory%601>对象，然后使用该对象来创建新的 WCF 客户端通道对象，每个调用。</span><span class="sxs-lookup"><span data-stu-id="c1601-111">Create a <xref:System.ServiceModel.ChannelFactory%601> object and then use that object to create new WCF client channel objects for each call.</span></span>  
  
 <span data-ttu-id="c1601-112">使用这些方法时要考虑的问题包括：</span><span class="sxs-lookup"><span data-stu-id="c1601-112">Issues to consider when using these approaches include:</span></span>  
  
-   <span data-ttu-id="c1601-113">如果服务正在使用会话维护特定于客户端的状态，然后你无法重用中间层 WCF 客户端与多个客户端层请求因为服务的状态与中间层客户端。</span><span class="sxs-lookup"><span data-stu-id="c1601-113">If the service is maintaining a client-specific state by using a session, then you cannot reuse the middle-tier WCF client with multiple-client tier requests because the service's state is tied to that of the middle-tier client.</span></span>  
  
-   <span data-ttu-id="c1601-114">如果服务必须在每个客户端基础上进行身份验证，必须创建而不是重用中间层 WCF 客户端 （或 WCF 客户端通道对象），因为在中间层上的每个传入请求的新客户端的中间层客户端凭据WCF 客户端后，无法修改 (或<xref:System.ServiceModel.ChannelFactory%601>) 已创建。</span><span class="sxs-lookup"><span data-stu-id="c1601-114">If the service must perform authentication on a per-client basis, you must create a new client for each incoming request on the middle tier instead of reusing the middle-tier WCF client (or WCF client channel object) because the client credentials of the middle tier cannot be modified after the WCF client (or <xref:System.ServiceModel.ChannelFactory%601>) has been created.</span></span>  
  
-   <span data-ttu-id="c1601-115">由于通道及其创建的客户端是线程安全的，因此它们可能不支持同时向网络中写入多条消息。</span><span class="sxs-lookup"><span data-stu-id="c1601-115">While channels and clients created by the channels are thread-safe, they might not support writing more than one message to the wire concurrently.</span></span> <span data-ttu-id="c1601-116">如果要发送较大的消息，特别是消息流，则发送操作可能会阻塞，因为它要等待另一个发送操作完成。</span><span class="sxs-lookup"><span data-stu-id="c1601-116">If you are sending large messages, particularly if streaming, the send operation might block waiting for another send to complete.</span></span> <span data-ttu-id="c1601-117">这会导致两种问题：缺少并发性；当控制流返回到重用信道的服务（即，共享的客户端调用了一种服务，该服务的代码路径会导致对共享的客户端进行回调）时，存在死锁的可能性。</span><span class="sxs-lookup"><span data-stu-id="c1601-117">This causes two sorts of problems: a lack of concurrency and the possibility of deadlock if the flow of control returns to the service reusing the channel (that is, the shared client calls a service whose code path results in a callback to the shared client).</span></span> <span data-ttu-id="c1601-118">这是 true 而不考虑你重复使用的 WCF 客户端的类型。</span><span class="sxs-lookup"><span data-stu-id="c1601-118">This is true regardless of the type of WCF client you reuse.</span></span>  
  
-   <span data-ttu-id="c1601-119">无论您是否共享出错的通道，您都必须处理它。</span><span class="sxs-lookup"><span data-stu-id="c1601-119">You must handle faulted channels regardless of whether you share the channel.</span></span> <span data-ttu-id="c1601-120">然而，重用通道时，出错通道可以关闭多个挂起的请求或发送。</span><span class="sxs-lookup"><span data-stu-id="c1601-120">When channels are reused, however, a faulting channel can take down more than one pending request or send.</span></span>  
  
 <span data-ttu-id="c1601-121">有关示例，演示重复使用多个请求的客户端的最佳做法，请参阅[ASP.NET 客户端中的数据绑定](../../../../docs/framework/wcf/samples/data-binding-in-an-aspnet-client.md)。</span><span class="sxs-lookup"><span data-stu-id="c1601-121">For an example that demonstrates best practices for reusing a client for multiple requests, see [Data Binding in an ASP.NET Client](../../../../docs/framework/wcf/samples/data-binding-in-an-aspnet-client.md).</span></span>  
  
 <span data-ttu-id="c1601-122">此外，如果客户端使用可用 <xref:System.Xml.Serialization.XmlSerializer> 进行序列化的数据类型，则会在运行时生成并编译这些数据类型的序列化代码，从而导致启动性能降低；但您可以提高那些客户端的启动性能。</span><span class="sxs-lookup"><span data-stu-id="c1601-122">In addition, you can increase the startup performance for those clients that use data types that are serializable using the <xref:System.Xml.Serialization.XmlSerializer> generate and compile serialization code for those data types at runtime, which can result in slow start-up performance.</span></span> <span data-ttu-id="c1601-123">[ServiceModel 元数据实用工具 (Svcutil.exe)](../../../../docs/framework/wcf/servicemodel-metadata-utility-tool-svcutil-exe.md)可以通过从应用程序的已编译程序集生成必要的序列化代码提高这些应用程序的启动性能。</span><span class="sxs-lookup"><span data-stu-id="c1601-123">The [ServiceModel Metadata Utility Tool (Svcutil.exe)](../../../../docs/framework/wcf/servicemodel-metadata-utility-tool-svcutil-exe.md) can improve start-up performance for these applications by generating the necessary serialization code from the compiled assemblies for the application.</span></span> <span data-ttu-id="c1601-124">有关详细信息，请参阅[How to： 改善启动时间的 WCF 客户端应用程序使用 XmlSerializer](../../../../docs/framework/wcf/feature-details/startup-time-of-wcf-client-applications-using-the-xmlserializer.md)。</span><span class="sxs-lookup"><span data-stu-id="c1601-124">For more information, see [How to: Improve the Startup Time of WCF Client Applications using the XmlSerializer](../../../../docs/framework/wcf/feature-details/startup-time-of-wcf-client-applications-using-the-xmlserializer.md).</span></span>
